## Context

前端是 React + Vite 应用，`@mantine/core` 和 `@mantine/hooks` 已经是现有依赖。入口页已经较多使用 Mantine，会议室页和 Agent 列表也已开始改造，但管理后台、直接加入、只读历史、纪要历史弹层和多个会议室子面板仍保留大量自定义 `button`、`input`、`textarea`、`select`、面板、徽标和弹窗样式。

这次变更是横跨多个前端页面的 UI 体系统一，不涉及后端 API、WebSocket、房间生命周期或数据模型。实现应以现有浅色、绿色强调、工作台式信息密度为基线，避免重新设计成营销页或引入新的视觉主题。

## Goals / Non-Goals

**Goals:**

- 使用 Mantine 组件统一主要页面的布局容器、按钮、表单、徽标、头像、列表、表格、弹窗和滚动区域。
- 让入口页、会议室页、管理后台、直接加入页、只读页和错误页共享一致的视觉层级、间距、控件尺寸和交互状态。
- 缩减项目自定义 CSS 中已被 Mantine 替代的通用控件样式，保留业务布局和特殊交互样式。
- 保持现有路由、数据加载、表单提交、房间连接、管理员操作、Agent 提及和纪要管理行为不变。

**Non-Goals:**

- 不新增或替换 UI 组件库。
- 不改造后端接口、数据库、WebSocket 消息结构或鉴权策略。
- 不改变页面信息架构和用户工作流。
- 不引入全新的主题系统、深色模式或品牌重设计。
- 不把所有 CSS 清零；复杂业务布局仍可使用局部 CSS。

## Decisions

1. **Mantine 作为默认 UI 原语**

   主要页面中的按钮、输入框、文本域、下拉框、徽标、头像、纸面容器、分组、堆叠、弹层、表格和滚动区域优先使用 Mantine 组件。这样可以减少不同页面自行维护控件尺寸、禁用态、边框、hover 和焦点样式的重复代码。

   备选方案是继续维护自定义 `.button`、`.panel`、`.text-input` 等样式，但这会延续当前页面间不一致的问题，且后续每个新页面都需要复制同一套视觉细节。

2. **保留应用级布局 CSS**

   `ChatRoom` 的三栏工作台、可拖拽面板宽度、消息气泡、提及自动补全、历史消息流、只读页网格等属于业务布局，不强行用 Mantine 完全替代。Mantine 组件负责控件和容器外观，自定义 CSS 负责固定比例、滚动区域、响应式约束和特殊定位。

   备选方案是全面迁移到 `AppShell` 或完全 CSS Modules 化，但当前项目没有 CSS Modules 约定，强制替换会扩大改动范围，也容易影响已稳定的房间交互。

3. **按页面族分批迁移**

   实施顺序从页面外壳和管理后台开始，再迁移会议室子面板，最后处理直接访问/只读/错误页和弹层。这样每一步都能通过构建和源代码测试验证，避免一次性大范围改动导致问题难定位。

   备选方案是先抽象统一布局组件再回填各页面，但当前目标是尽量使用现成组件库，新增项目级抽象只有在明显减少重复时才需要。

4. **测试聚焦结构约束与现有行为**

   现有 Node 测试覆盖会议纪要、消息来源、角色集、房间访问、复制等行为。本变更应补充轻量源代码测试，验证关键页面使用 Mantine 组件并且不回退到主要自定义控件，同时继续运行现有相关测试和生产构建。

## Risks / Trade-offs

- Mantine 组件默认样式可能与现有 CSS 选择器互相覆盖 → 迁移时逐页检查类名作用域，删除或收窄过宽的通用选择器。
- 一次性替换大量表单控件可能破坏受控输入、提交禁用态或焦点行为 → 保留现有状态管理，只替换渲染层，并用现有测试和构建验证。
- 管理后台列表和会议详情中存在密集信息展示 → 使用 `Table`、`Stack`、`Group`、`Badge` 等组件保持扫描效率，不改成大卡片堆叠。
- 会议室的拖拽布局和滚动区域对尺寸敏感 → 先保留核心布局 CSS，仅替换内部控件和面板外观。
