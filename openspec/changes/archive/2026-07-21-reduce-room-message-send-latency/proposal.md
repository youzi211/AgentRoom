## Why

会议房间当前会在广播用户消息前同步执行焦点 LLM 分析，导致用户按下 Enter 后必须等待模型调用完成才能看到自己的消息。焦点分析属于派生能力，不应阻塞已成功持久化消息的实时交付，也不应因失败或空结果让后续消息持续重复承受分析延迟。

## What Changes

- 用户消息成功持久化后立即通过房间实时通道广播，不再等待焦点分析完成。
- 将焦点分析移出用户消息交付关键路径，通过有界的后台执行机制处理并单独发布焦点更新。
- 焦点分析尝试完成后推进该房间的分析游标，包括模型调用失败、响应解析失败或结果为空的情况，避免同一批消息被后续发送重复触发。
- 保持消息可靠性语义不变：持久化失败的消息不广播为成功，现有 WebSocket 消息格式和前端展示协议不变。
- 本次不增加前端乐观消息、不调整消息列表渲染，也不改变 Agent 回复调度逻辑。

## Capabilities

### New Capabilities

- `realtime-message-delivery`: 定义持久化用户消息的低延迟广播顺序、焦点分析异步隔离以及分析游标在失败和空结果下的推进规则。

### Modified Capabilities

无。

## Impact

- 后端服务层：`backend/internal/service/realtime_session.go`、`room_commands.go`、`focus_service.go` 及相关服务装配代码。
- 后端测试：补充用户消息广播不受焦点 LLM 阻塞、焦点更新异步发布、失败/空结果不重复分析的回归测试。
- 实时协议：继续使用现有 `message` 和 `focus_update` 事件，不修改客户端事件结构。
- 数据与部署：不涉及数据库 schema、环境变量、外部依赖或部署方式变更。
