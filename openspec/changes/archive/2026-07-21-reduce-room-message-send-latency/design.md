## Context

当前 WebSocket 处理链路在接收用户消息后依次执行消息持久化、房间内存追加、焦点分析和消息广播。焦点分析每累计一定数量的消息调用一次 LLM，因此模型网络延迟直接进入用户消息的可见延迟；分析失败或返回空结果时又不会推进 `lastAnalyzed`，后续消息会持续满足触发条件。

消息本体是需要可靠持久化和实时广播的核心数据，焦点是可重建、允许稍后到达的派生数据。现有客户端已经分别处理 `message` 与 `focus_update` 事件，因而可以在不修改协议的前提下拆分两者的完成时序。

## Goals / Non-Goals

**Goals:**

- 将用户消息的可见延迟限制在持久化和 WebSocket 广播路径内，不再包含焦点 LLM 调用。
- 使用有界、按房间去重的后台调度执行焦点分析，防止慢模型造成无界 goroutine 或重复并发分析。
- 无论焦点分析成功、失败、解析失败还是返回空结果，都准确推进本次快照对应的分析游标。
- 保持现有消息可靠性、房间内存状态和实时事件协议不变。

**Non-Goals:**

- 不实现客户端乐观消息或 pending/重试 UI。
- 不改变消息数据库 schema、WebSocket 事件字段或前端消息渲染方式。
- 不调整焦点提示词、触发阈值、模型解析规则或 Agent 回复队列。
- 不持久化焦点状态；焦点仍是进程内派生能力。

## Decisions

### 1. 消息持久化成功后先广播，再调度焦点分析

`HandleHumanMessage` 只负责校验、持久化和追加房间消息，不再等待焦点 LLM。`PostRealtimeMessage` 在这些步骤成功后立即广播现有 `message` 事件，然后把该消息交给焦点后台调度器。

持久化失败时仍返回错误，不追加房间消息、不广播成功事件，也不调度焦点分析。这保留了当前“服务端持久化结果是成功确认”的语义。

备选方案是在持久化前广播，虽然更快，但会让客户端看到可能无法恢复的消息；本次不采用。前端乐观更新同样需要客户端消息 ID、确认和去重协议，超出本次范围。

### 2. 焦点分析使用独立的有界调度器并按房间合并工作

焦点分析使用独立于 Agent 回复的固定 worker 和有界队列。消息观察只做短暂的内存状态更新；达到阈值时记录目标消息计数并尝试调度，不在 WebSocket 处理 goroutine 中执行模型调用或等待队列容量。

同一房间最多存在一个 queued/in-flight 分析。分析期间到达的新消息继续计入房间状态，但不会启动并发分析；完成后若新增消息再次达到阈值，则基于最新快照安排下一次工作。队列压力下合并同一房间的待处理工作，而不是为每条消息创建 goroutine。

后台调用使用脱离 WebSocket 请求取消信号、但带有限超时的 context，避免客户端断开取消已接收消息的派生处理，也避免单次模型调用永久占用 worker。调度和超时只写结构化日志，不向客户端发送消息失败事件。

备选方案是直接 `go focus.AddMessage(...)`，实现较少但并发无界且难以保证同一房间的游标顺序，因此不采用。复用 Agent 回复队列会让两个不同优先级的派生任务相互争用，也不采用。

### 3. 分析游标表示“已完成尝试的消息位置”

每个分析任务捕获不可变的 `targetCount`。任务进入终态后，无论得到有效焦点、模型错误、超时、无效 JSON 或空数组，均将 `lastAnalyzed` 单调推进到该 `targetCount`，并清除该房间的 in-flight 状态。

只有有效且非空的结果替换焦点集合并发布 `focus_update`。失败和空结果不发布焦点更新，但也不允许下一条消息立即重试同一批内容；只有 `targetCount` 之后新累计的消息再次达到阈值才触发新分析。迟到结果不得把游标或焦点状态回退到更旧的目标位置。

### 4. 保持实时协议兼容

继续使用现有 `message` 和 `focus_update` 事件。客户端先收到用户消息，焦点更新在后台分析成功后独立到达；无需新增字段或前端兼容分支。

## Risks / Trade-offs

- [焦点更新相对消息更晚到达] -> 这是解除关键路径阻塞的预期结果，客户端已经独立处理两类事件。
- [高负载时后台队列积压] -> 对同一房间合并工作、限制 queued/in-flight 数量，并让消息广播完全不等待队列。
- [进程退出会丢失未完成的焦点任务] -> 焦点是可重建派生状态，已持久化消息不受影响；本次不引入新的持久化表。
- [异步结果可能乱序] -> 每个任务携带 `targetCount`，游标单调推进且同一房间禁止并发分析。
- [模型调用持续占用 worker] -> 为后台焦点调用设置有限超时，并在所有终态清理调度状态。

## Migration Plan

1. 先以服务层单元测试锁定持久化、广播和焦点调度顺序。
2. 引入焦点后台调度和游标终态规则，保持现有事件结构。
3. 运行完整 Go 测试、静态检查和构建；无需数据库迁移或前端发布配套改动。
4. 回滚时恢复原同步调用路径即可，消息和数据库 schema 不需要回滚。

## Open Questions

无。worker 数量、队列容量和超时采用包内常量，并由测试验证有界性；本次不增加部署配置面。
