# AgentRoom 功能梳理与需求文档

更新时间：2026-06-17

## 1. 文档目的

本文档基于当前仓库实现整理 AgentRoom 已具备的产品能力、业务规则和下一步演进方向。它不是全新 PRD，也不描述尚未落地的宏大愿景；它的目标是让后续开发者能从当前代码事实出发继续推进。

## 2. 产品定位

AgentRoom 是一个实时文本会议工作台。人类参与者和可配置的角色型 Agent 在同一个房间协作，Agent 默认保持沉默，只有被明确 `@AgentName` 提及时才进入讨论。

核心价值：

- 把 AI Agent 从独立问答窗口转为会议中的按需参与者。
- 通过显式 `@Agent` 机制，让 Agent 介入可控、可解释、可回溯。
- 将会议讨论、知识文档、焦点事项、纪要和历史记录放进同一条工作流。
- 为管理员提供会后治理能力，包括会议管理、纪要管理、Agent 管理和知识文档管理。

## 3. 当前用户角色

| 角色 | 定义 | 主要能力 |
| --- | --- | --- |
| 普通参会者 | 通过显示名加入会议的人类用户 | 创建会议、加入会议、实时发言、`@Agent`、查看焦点与纪要、在会议关闭后只读回看 |
| 当前房主 | 当前在线的人类参会实例，由系统在运行中指定 | 关闭会议、转交房主权限 |
| 管理员 | 持有后台管理密钥的运营或管理角色 | 查看全部会议、管理生命周期、查看完整历史、管理纪要版本、管理 Agent 与知识文档 |
| Agent | 会议中的系统角色参与者 | 基于角色设定、历史消息和知识上下文生成回复 |

## 4. 核心业务对象

| 对象 | 说明 |
| --- | --- |
| Room | 会议基本容器，包含名称、状态、参与 Agent、消息历史和权限信息 |
| Participant | 某次在线加入会议的人类实例，当前版本未引入正式账号体系 |
| Agent | 可配置的角色型智能参与者，具备名称、职责、描述、提示词和启用状态 |
| Role Template | 静态产品默认角色模板，用于加速创建常见会议 Agent |
| Role Set | 创建会议时的角色选择快捷方式，不是持久化会议模板 |
| Message | 会议中的人类消息、Agent 消息或系统消息 |
| Knowledge Document | 上传到会议或 Agent 范围内的 Markdown 文档 |
| Knowledge Chunk | Markdown 文档切分后的检索片段，保留文档来源名称 |
| Focus Point | 系统从会议讨论中提取的重点话题或决策焦点 |
| Meeting Minutes | 可生成、可持久化、可导出的会议纪要内容，支持多版本 |

## 5. 核心业务流程

### 5.1 创建会议

用户输入显示名后可以创建新会议，并在创建时完成以下设置：

- 设置会议名称。
- 选择本次会议启用的 Agent。
- 使用推荐角色组快速选择常见 Agent 组合。
- 可选设置房间口令。
- 选择 Agent 对话模式。

角色组只是选择快捷方式，用户仍可以手动勾选或取消任意 Agent。

### 5.2 加入会议

用户可通过会议 ID 加入现有会议，并在需要时输入房间口令。房间访问参数会保存在浏览器会话中，便于刷新后恢复。

### 5.3 实时协作

进入会议后，参会者可以：

- 通过 WebSocket 参与实时文本讨论。
- 查看当前在线参与者和已加入会议的 Agent。
- 通过 `@Agent` 点名某个 Agent 发言。
- 查看焦点事项、Agent 活动状态、会议纪要预览和知识文件。
- 在 Agent 回复下方看到本次回复引用过的知识来源文件。

### 5.4 会中知识沉淀

系统支持两类 Markdown 知识输入：

- 会议级知识：所有参与该会议的 Agent 均可参考。
- Agent 级知识：仅对应 Agent 发言时参考。

上传后系统会解析 Markdown 文本、切分为可检索片段，并在 Agent 被触发时按当前问题检索相关片段注入上下文。片段会携带 `documentId` 和 `documentName`，用于提示词来源标签和消息来源展示。

### 5.5 会议结束与会后管理

会议可以由房主主动关闭，也可以在最后一位人类参会者离开后自动关闭。关闭后的会议仍可通过原链接只读回看；归档后的会议只允许管理员访问。

## 6. 功能需求

### 6.1 会议入口与创建

- 首页同时提供“创建会议”和“加入会议”两条路径。
- 创建会议时必须填写显示名。
- 创建会议时可选择启用的 Agent 列表。
- 创建会议时可使用后端提供的推荐角色组快速选择 Agent。
- 创建会议时可设置房间口令；未设置时会议为开放访问。
- 创建会议时可选择对话模式：
  - `mention_fanout`：点名单轮。
  - `guided_dialogue`：引导多轮。
- 加入会议时需要输入显示名和会议 ID。

### 6.2 实时会议室

- 通过 WebSocket 建立会议实时连接。
- 实时同步房间快照、参与者变化、消息流、焦点更新和 Agent 活动事件。
- 会议室展示内容至少包括会议名称、会议 ID、连接状态、参与者列表、Agent 列表、消息列表、输入区、焦点、活动、纪要和知识文件。
- 参会者发送消息后，应立即广播给所有在线用户。
- 房间消息历史应在进入会议时拉取，并与实时流合并展示。

### 6.3 Agent 协作机制

- Agent 默认保持沉默，不对普通消息自动发言。
- Agent 仅在被明确 `@名称` 提及时触发。
- 一条消息中提及多个 Agent 时，多个 Agent 可以依次响应。
- Agent 回复结合房间近期消息、当前参与者、房间 Agent、会议级知识片段和 Agent 专属知识片段。
- Agent 回复失败时，系统应在房间中写入可见的系统提示。
- 系统应清理模型响应中的 `<think>` / `<thinking>` 私有推理内容。
- Agent 回复应保存确定性的知识来源摘要，而不是要求模型自行生成引用。

### 6.4 Agent 对话模式

当前支持两种对话模式：

- `mention_fanout`：用户提及哪个 Agent，就由对应 Agent 回复。
- `guided_dialogue`：被用户点名的 Agent 先回复，之后系统按策略调度其他 Agent 接力发言。

`guided_dialogue` 需要支持最大自治轮次、单个 Agent 最大发言次数、自我跟进、Agent 互相点名和轮次冷却等限制，并在达到上限、重复、空回复、超时或无可选发言者时主动停止。

### 6.5 知识库能力

- 仅支持上传 Markdown 文件。
- 支持扩展名 `.md` 和 `.markdown`。
- 单文件大小限制为 1MB。
- 上传后抽取文本内容并切分为可检索片段。
- 检索仍基于现有文本片段搜索，不引入向量数据库。
- Agent 提示词中的知识片段应带有来源标签，例如 `[room: roadmap.md #1]`。
- Agent 消息返回时可附带 `knowledgeSources`，前端以紧凑来源标签展示。
- 管理员可删除已上传的知识文档。

### 6.6 角色模板与角色组

- 后端通过 `GET /api/agent-templates` 提供静态角色模板。
- 初始模板包括产品经理、架构师、质量评审、风险评审和会议纪要员。
- 模板用于预填 Agent 创建表单，保存前仍允许编辑。
- 后端通过 `GET /api/agent-role-sets` 提供静态会议角色组。
- 角色组只用于创建会议时快速勾选 Agent，不新建会议模板子系统，也不持久化选择规则。

### 6.7 焦点事项与活动可观测性

- 系统可从会议消息中提取焦点事项。
- 会议界面以列表或时间线方式展示焦点事项。
- 系统记录并展示 Agent run 与 dialogue run 活动，包括开始、完成、状态、触发消息、相关 Agent 和轮次。
- 管理端和前台都应能基于历史记录查看近期 Agent 活动。

### 6.8 会议纪要

- 用户可在会议中触发纪要生成。
- 系统优先调用后端生成并持久化纪要。
- 当后端生成不可用时，前端可回退到本地草稿预览。
- 用户可将纪要导出为 Markdown。
- 管理员可查看纪要历史版本、重新生成、手动编辑并保存新版本。
- 关闭后的会议允许普通用户查看最新已保存纪要，但不能新增纪要版本。

### 6.9 生命周期与权限

会议生命周期分为：

| 状态 | 含义 |
| --- | --- |
| `active` | 会议进行中，可实时加入和发言 |
| `closed` | 会议已结束，普通用户可只读回看 |
| `archived` | 会议已归档，仅管理员可访问 |

规则：

- 第一位成功进入 active 会议的人类参会者自动成为当前房主。
- 当前房主可以转交房主权限或主动关闭会议。
- 最后一位人类参会者离开时，系统启动 30 秒宽限期；无人重新进入则自动转为 `closed`。
- 管理员可归档、重开或取消归档会议。
- 取消归档必须回到 `closed`，不能直接回到 `active`。

## 7. 非功能与系统约束

- 后端：Go / Gin / gorilla/websocket。
- 前端：React + Vite。
- 数据库：MySQL。
- LLM 接入：OpenAI 兼容 Chat Completions。
- 当前仍是单体轻量部署，适合阶段性协作场景。
- 未配置 LLM API Key 时，人类会议流程仍应可用；Agent 回复会失败并以系统消息提示。
- UI 与文档必须保持可读 UTF-8，不允许出现乱码文案。

## 8. 当前范围边界

当前版本不包含：

- 正式登录、账号、组织或租户体系。
- PDF、Word、图片等非 Markdown 知识格式。
- 向量数据库或 embedding 检索。
- 音视频会议能力。
- 多实例实时同步、Redis Pub/Sub 或队列化 Agent 执行。
- 知识来源点击打开文档预览。
- 持久化会议模板或复杂议程模板。

## 9. 下一步建议

优先级建议：

1. 保持现有架构边界，继续强化 `KnowledgeService`、`AgentService`、`RoomService`，不要另起平行系统。
2. 为知识来源芯片增加可点击的文档预览，但仍复用现有知识文档 API。
3. 为新建 Agent 保存可选 `templateID`，让角色组匹配不再依赖名称。
4. 设计用户账号和会议权限模型，再决定管理员能力如何下沉到普通团队空间。
5. 在文本检索瓶颈真实出现后，再评估 hybrid search 或向量库。
