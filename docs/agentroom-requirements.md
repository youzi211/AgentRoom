# AgentRoom 功能梳理与需求文档（基于当前实现）

更新时间：2026-06-16

## 1. 文档目的

本文档用于梳理 AgentRoom 当前版本已经具备的产品能力，并将其整理为一份可用于需求沟通、版本说明和后续迭代评审的需求文档。

说明：

- 本文档以当前仓库实现为准，属于“基于现状反向整理”的需求说明。
- 文档重点描述用户可感知能力、业务规则、角色权限和关键边界。
- 文档不展开底层实现细节，但会保留必要的系统约束，方便后续继续演进。

## 2. 产品定位

AgentRoom 是一个面向多人协作场景的实时文本会议空间。系统把人类参会者、可配置的角色型 Agent、会议知识资料、焦点追踪、会议纪要和会后历史沉淀放进同一套会议工作流中。

它不是一个“一问一答”的普通聊天机器人，也不是视频会议系统，而是一个让人和 Agent 在同一个会议上下文中协作的工作台。

### 2.1 核心价值

- 把 AI Agent 从独立问答窗口，转为会议中的按需参与者。
- 通过显式 `@Agent` 机制，让 Agent 的介入可控、可解释、可回溯。
- 将会议讨论、知识文件、焦点事项、纪要和历史记录串联起来。
- 为管理员提供会后治理能力，包括会议管理、纪要管理和 Agent 管理。

## 3. 用户角色

| 角色 | 定义 | 主要能力 |
| --- | --- | --- |
| 普通参会者 | 通过显示名加入会议的人类用户 | 创建会议、加入会议、实时发言、`@Agent`、查看焦点与纪要、在会议关闭后只读回看 |
| 当前会议房主 | 当前在线的人类参会者实例，由系统在运行中指定 | 关闭会议、转交房主权限 |
| 管理员 | 持有后台管理权限的运营/管理角色 | 查看全部会议、管理会议生命周期、查看完整历史、管理纪要版本、管理 Agent 与知识库 |
| Agent | 会议中的系统角色参与者 | 基于角色设定、历史消息和知识上下文进行响应 |

## 4. 核心业务对象

| 对象 | 说明 |
| --- | --- |
| 会议室 Room | 会议的基本容器，包含名称、状态、参与 Agent、消息历史、权限信息等 |
| 参与者 Participant | 某次在线加入会议的人类实例，当前版本未引入正式账号体系 |
| Agent | 可配置的角色型智能参与者，具备名称、职责、描述、角色模板、启用状态 |
| 消息 Message | 会议中的人类消息、Agent 消息或系统消息 |
| 知识文档 Knowledge Document | 上传到会议或 Agent 范围内的 Markdown 文档 |
| 焦点事项 Focus Point | 系统从会议讨论中提取出的重点话题或决策焦点 |
| 会议纪要 Meeting Minutes | 可生成、可持久化、可导出的会议纪要内容，支持多版本 |

## 5. 核心业务流程

### 5.1 创建会议

用户输入显示名后可创建新会议，并在创建时完成以下设置：

- 设置会议名称。
- 选择本次会议启用的 Agent。
- 可选设置房间口令。
- 选择 Agent 对话模式。

创建成功后，用户会直接进入实时会议室。

### 5.2 加入会议

用户可通过会议 ID 加入现有会议，并在需要时输入房间口令。

### 5.3 实时协作

进入会议后，参会者可以：

- 通过 WebSocket 参与实时文本讨论。
- 查看当前在线参会者和已加入会议的 Agent。
- 通过 `@Agent` 方式点名某个 Agent 发言。
- 查看焦点事项、Agent 活动状态和会议纪要预览。

### 5.4 会中沉淀

会议进行过程中，系统支持：

- 上传会议级知识资料。
- 为 Agent 绑定专属知识资料。
- 自动抽取会议焦点。
- 生成并导出会议纪要。

### 5.5 会议结束与会后管理

会议可以被房主主动关闭，也可以在最后一位人类参会者离开后自动关闭。关闭后的会议仍可通过原链接进行只读回看；归档后的会议只允许管理员访问。

## 6. 功能需求

### 6.1 会议入口与创建

系统应支持以下入口能力：

- 首页同时提供“创建会议”和“加入会议”两条主路径。
- 创建会议时必须填写显示名。
- 创建会议时可选择本次启用的 Agent 列表。
- 创建会议时可设置房间口令；未设置时，会议为开放访问。
- 创建会议时可选择对话模式：
  - `mention_fanout`：点名单轮。
  - `guided_dialogue`：引导多轮。
- 加入会议时需要输入显示名和会议 ID。
- 房间访问会在浏览器会话内保留基础进入信息，便于页面刷新后恢复。

### 6.2 实时会议室

系统应支持以下实时协作能力：

- 通过 WebSocket 建立会议实时连接。
- 实时同步房间快照、参与者变化、消息流、焦点更新和 Agent 活动事件。
- 会议室展示内容至少包括：
  - 会议名称与会议 ID
  - 当前连接状态
  - 参与者列表
  - Agent 列表
  - 消息列表
  - 消息输入区
  - 会议上下文、焦点、活动、纪要、知识文件侧栏
- 参会者发送消息后，应立即广播给所有在线用户。
- 房间消息历史应在进入会议时拉取，并与实时流合并展示。
- 用户可复制会议 ID 以便分享给其他参会者。

### 6.3 Agent 协作机制

系统中的 Agent 应满足以下行为规则：

- Agent 默认保持沉默，不应对普通消息自动发言。
- Agent 仅在被明确 `@名称` 提及时触发。
- 一条消息中提及多个 Agent 时，多个 Agent 可以依次响应。
- Agent 回复应结合以下上下文：
  - 房间近期可见消息
  - 当前会议参与者
  - 房间内已选 Agent
  - 会议级知识片段
  - Agent 专属知识片段
- Agent 响应失败时，系统应在房间中写入可见的系统提示，而不是静默失败。
- 系统应清理模型响应中的私有推理标签或不可对用户展示的思考内容。

### 6.4 Agent 对话模式

系统当前支持两种对话模式：

#### 6.4.1 点名单轮模式 `mention_fanout`

- 用户提及哪个 Agent，就由哪个 Agent 回复。
- 在允许 Agent-to-Agent 提及时，Agent 的回复也可以继续触发其他 Agent。
- 系统需要限制单个 Agent 连续自我跟进和总自治轮次，避免无限循环。

#### 6.4.2 引导多轮模式 `guided_dialogue`

- 被用户点名的 Agent 先回复。
- 之后系统可根据消息中的提及关系继续调度其他 Agent 接力发言。
- 系统需要支持以下限制参数：
  - 最大自治轮次
  - 单个 Agent 最大发言次数
  - 是否允许自我跟进
  - 是否允许 Agent 点名其他 Agent
  - 轮次冷却时间
- 系统需要记录完整的 dialogue run，用于后续审计和展示。
- 当出现空回复、重复回复、超时、无可继续发言者或上限触达时，系统应主动停止对话链。

### 6.5 知识库能力

系统应支持两类知识输入：

- 会议级知识：所有参与该会议的 Agent 均可参考。
- Agent 级知识：仅特定 Agent 发言时参考。

知识库功能要求如下：

- 仅支持上传 Markdown 文件。
- 支持的扩展名为 `.md` 或 `.markdown`。
- 单个文件大小限制为 1MB。
- 上传后系统应抽取文本内容并切分为可检索片段。
- Agent 在生成回复时，应按当前问题检索相关知识片段并注入上下文。
- 管理员可以删除已上传的知识文档。

### 6.6 焦点事项与活动可观测性

系统应提供会议过程的辅助观察能力：

- 自动从会议消息中提取焦点事项。
- 在会议界面中以时间线或列表方式展示焦点事项。
- 记录并展示 Agent 运行活动，包括：
  - `agent_run`
  - `dialogue_run`
- 活动展示应覆盖开始、完成、状态、触发消息、涉及 Agent 等信息。
- 管理端和前台都应能基于历史记录查看近期 Agent 活动。

### 6.7 会议纪要

系统应支持会议纪要的生成、保存、版本管理和导出。

普通会议界面的能力：

- 用户可在会议中触发纪要生成。
- 系统应优先调用后端生成并持久化纪要。
- 当后端生成不可用时，前端可回退到本地草稿预览，保证纪要面板可用。
- 用户可将纪要导出为 Markdown 文件。

管理员的纪要能力：

- 管理员可查看会议纪要历史版本。
- 管理员可重新生成新的 AI 纪要版本。
- 管理员可手动编辑纪要内容，并保存为新的手动版本。
- 管理员可导出当前纪要内容为 Markdown 文件。

会后只读能力：

- 关闭后的会议，普通用户可查看最新已持久化的纪要。
- 关闭后的会议，普通用户只能下载/查看，不可新增纪要版本。

### 6.8 会议生命周期与房主机制

系统应将会议生命周期拆分为三个状态：

| 状态 | 含义 |
| --- | --- |
| `active` | 会议进行中，可实时加入和发言 |
| `closed` | 会议已结束，普通用户可只读回看 |
| `archived` | 会议已归档，仅管理员可访问 |

房主与关闭规则：

- 第一位成功进入 active 会议的人类参会者自动成为当前房主。
- 当前房主可以把房主权限转交给另一位在线人类参会者。
- 当前房主可以主动关闭会议。
- 当最后一位人类参会者离开时，系统启动 30 秒宽限期。
- 宽限期内若无人重新进入，会议自动转为 `closed`。
- 房主身份与当前在线参与实例绑定，不是长期账号身份。

### 6.9 关闭、归档与恢复规则

系统应支持以下生命周期操作：

- 房主手动关闭：`active -> closed`
- 最后一位人类离开并超时关闭：`active -> closed`
- 管理员重开会议：`closed -> active`
- 管理员归档会议：`active -> archived` 或 `closed -> archived`
- 管理员取消归档：`archived -> closed`

业务要求：

- 关闭会议后，实时连接应终止，在线用户退出 live 会话。
- 归档 active 会议时，也应立即终止实时会话。
- 取消归档后，会议必须回到 `closed`，不能直接回到 `active`。
- 普通用户不能自行重开会议。

### 6.10 只读回看能力

关闭后的会议应支持只读回看：

- 普通用户可通过原会议链接查看会议基本信息。
- 普通用户可查看历史消息列表。
- 普通用户可分页加载更早消息。
- 普通用户可查看最新已保存纪要。
- 普通用户不可重新加入 WebSocket。
- 普通用户不可发送消息。
- 普通用户不可转交房主或恢复会议。

归档后的会议不对普通用户开放，只允许管理员通过后台查看。

### 6.11 管理后台

系统应提供后台管理控制台，至少包含两个主要模块：

#### 6.11.1 会议管理

- 查看全部会议列表。
- 按状态筛选会议：全部、进行中、已关闭、已归档。
- 查看会议基础信息：
  - 名称
  - ID
  - 状态
  - 消息数
  - 创建时间
  - 最近消息时间
- 进入会议详情，查看：
  - 会议概览
  - 完整消息历史
  - 最新纪要
  - 纪要版本历史
- 执行会议生命周期操作：
  - 归档
  - 重开
  - 取消归档

#### 6.11.2 Agent 管理

- 查看全部 Agent 列表。
- 创建 Agent。
- 编辑 Agent 的名称、角色标签、职责描述、角色模板、启用状态。
- 删除 Agent。
- 启停 Agent，使其是否出现在新建会议可选列表中。
- 管理 Agent 专属知识文档。

## 7. 权限与访问矩阵

### 7.1 房间状态权限矩阵

| 场景 | `active` | `closed` | `archived` |
| --- | --- | --- | --- |
| 普通用户查看会议元数据 | 支持 | 支持 | 不支持 |
| 普通用户查看历史消息 | 支持 | 支持 | 不支持 |
| 普通用户查看/下载最新纪要 | 支持 | 支持 | 不支持 |
| 普通用户加入实时 WebSocket | 支持 | 不支持 | 不支持 |
| 普通用户发送消息 | 支持 | 不支持 | 不支持 |
| 普通用户触发纪要生成 | 支持 | 不支持 | 不支持 |
| 管理员查看会议 | 支持 | 支持 | 支持 |
| 管理员管理纪要版本 | 支持 | 支持 | 支持 |
| 管理员归档会议 | 支持 | 支持 | 不支持 |
| 管理员重开会议 | 不适用 | 支持 | 不支持 |
| 管理员取消归档 | 不适用 | 不适用 | 支持，且恢复到 `closed` |

### 7.2 口令与后台密钥

- 房间可设置口令，保护会议元数据、消息历史、纪要读取和 WebSocket 加入。
- 管理员在持有有效后台密钥时，可绕过单个房间口令进行管理操作。
- Agent 管理、知识上传删除、会议列表等后台能力应受管理员密钥保护。

## 8. 非功能与系统约束

### 8.1 技术形态

- 后端：Go
- 前端：React + Vite
- 实时通信：WebSocket
- 数据库：MySQL
- LLM 接入：OpenAI 兼容接口

### 8.2 持久化要求

系统当前需要持久化以下关键数据：

- Agent 配置
- 会议元数据与状态
- 会议内 Agent 快照
- 参与者记录
- 消息历史
- Agent run / dialogue run
- 知识文档与文本切片
- 会议纪要版本

### 8.3 运行约束

- 未配置 LLM API Key 时，系统仍应允许人类会议流程继续运行，但 Agent 回复会失败并以房间内系统消息提示。
- 数据库迁移可在配置允许时自动执行。
- 房间实时状态当前基于单服务运行时管理，适合当前阶段的轻量协作场景。

## 9. 当前范围边界与已知限制

以下内容不属于当前版本完整能力，或仍存在明显边界：

- 当前没有正式登录、账号体系、组织体系。
- 房主身份不是长期身份，而是与当前在线参与实例绑定。
- 普通用户不能编辑历史消息，也不能编辑历史纪要版本。
- 知识库目前只支持 Markdown，不支持 PDF、Word、图片等格式。
- 知识检索仍以当前文本切片检索为主，未引入向量数据库能力。
- 项目核心场景是文本会议，不包含音视频会议能力。
- 后端当前更适合单体/轻量部署场景，大规模多实例协同仍需额外架构增强。
